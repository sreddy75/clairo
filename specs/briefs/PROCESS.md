# SME → Spec: Async Requirements Pipeline

## The Problem

SMEs (Unni, Vik, domain experts) know what needs building but don't have time to write formal specs. Engineers need structured specs to build the right thing. The gap between "I need a thing" and "here's a spec" currently requires synchronous meetings and manual translation.

## The Pipeline

```
SME (10-15 min)          You/Asaf (async)           Engineering (async)
─────────────           ─────────────────          ──────────────────
                                                    
/sme-capture     →     /brief-to-spec      →      /speckit.plan
  Conversational          Transforms brief            Technical design
  brain dump              into spec.md                + API contracts
  in Cowork               (speckit format)            
                                                    /speckit.tasks
  Output:                 Output:                     Task breakdown
  specs/briefs/           specs/###-name/             for implementation
  YYYY-MM-DD-name.md      spec.md                   
                                                    /speckit.implement
                                                      TDD execution
```

## How It Works

### Step 1: SME Capture Session (SME — 10-15 min)

The SME opens a Cowork session and says `/sme-capture [topic]`.

Claude runs a conversational discovery — no forms, no templates. It asks smart follow-up questions across 7 dimensions: problem, users, happy path, sad paths, data, boundaries, success. The SME talks naturally.

Output: A structured brief saved to `specs/briefs/`.

**Tips for SMEs:**
- Don't prep. Just talk about what you need.
- Concrete examples > abstract descriptions. "Like when Vik had that BAS client with 50 uncoded transactions" is gold.
- Say "I don't know" if you don't know. Gaps get noted and filled later.
- 10 minutes is fine. 15 is plenty. Don't force it.

### Step 2: Brief → Spec Conversion (Product/Eng — async)

Someone (Suren, Asaf, or even Claude autonomously) runs `/brief-to-spec [brief-file]`.

Claude reads the brief, cross-references it against Clairo's domain knowledge, existing specs, and the constitution, then generates a full `spec.md` in speckit format — user stories with priorities, functional requirements, audit checklist, success criteria, the works.

Assumptions are marked with 💡 so reviewers can spot them fast.

Output: `specs/###-feature-name/spec.md` + quality checklist.

### Step 3: Review & Clarify (async)

Quick review of the generated spec. Two paths:
- **Clean**: Spec looks good → proceed to `/speckit.plan`
- **Has questions**: Run `/speckit.clarify` to resolve max 3 unknowns (can loop back to SME if needed)

### Step 4: Standard Speckit Flow (Engineering — async)

From here it's the normal pipeline:
1. `/speckit.plan` — Technical architecture, data models, API contracts
2. `/speckit.tasks` — Dependency-ordered, independently testable task list
3. `/speckit.implement` — TDD execution

## When to Use What

| Situation | Do This |
|-----------|---------|
| SME has a feature idea | `/sme-capture` → `/brief-to-spec` → speckit |
| SME has a bug report or complaint | `/sme-capture` (shorter session) → evaluate if it's a spec or a ticket |
| You already know what to build | Skip capture, write brief manually or go straight to `/speckit.specify` |
| SME sends a voice note or Slack message | Copy the content into a brief template manually, then `/brief-to-spec` |
| Feature needs splitting | `/brief-to-spec` will flag this — it checks scope sizing |

## Quality Guarantees

The `/brief-to-spec` skill enforces the same quality standards as `/speckit.specify`:
- User stories are prioritized P1/P2/P3 and independently testable
- Functional requirements use MUST/SHOULD language and are testable
- Success criteria are measurable and technology-agnostic
- Audit checklist is filled (not boilerplate) — critical for Clairo's tax domain
- Max 3 [NEEDS CLARIFICATION] markers
- Constitution compliance checked

## FAQ

**Q: Can the SME do this without me?**
A: Yes. They open Cowork, say `/sme-capture`, talk for 10 minutes, done. The brief is saved. You convert it later.

**Q: What if the brief is too thin?**
A: `/brief-to-spec` will tell you. It checks completeness before generating and flags gaps. You can either fill them yourself or loop back to the SME.

**Q: Can I edit the brief before converting?**
A: Absolutely. The brief is just a markdown file. Edit it, add context, then run `/brief-to-spec`.

**Q: Does this replace speckit?**
A: No. This is a **front door** to speckit. It handles the messy "requirements gathering" phase that speckit assumes is already done. The output feeds directly into the existing speckit pipeline.

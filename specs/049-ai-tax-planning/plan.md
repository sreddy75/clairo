# Implementation Plan: AI Tax Planning & Advisory

**Branch**: `049-ai-tax-planning` | **Date**: 2026-03-31 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/049-ai-tax-planning/spec.md`

## Summary

Build a tax planning module that pulls a client's P&L from Xero (or accepts manual entry), calculates the estimated Australian tax position for companies, individuals, trusts, and partnerships, and provides an AI-powered chat interface for modelling what-if tax scenarios with deterministic calculations. The module is fully additive — a new `backend/app/modules/tax_planning/` module with a corresponding frontend tab in the client detail page. The AI agent uses Claude tool-use to call a pure-function tax calculator, ensuring numerically accurate before/after comparisons. Tax rates are stored in a database table for update without code deployment. Plans export as branded PDF documents.

**Testing**: pytest + pytest-asyncio (backend), manual verification (frontend Phase 1)
**Target Platform**: Web application (accountant dashboard)
**Project Type**: Web — backend API + frontend SPA
**Performance Goals**: Client-to-tax-position in <2 minutes (SC-001), AI scenario response in <15 seconds (SC-002), tax calculations within $1 accuracy (SC-003)
**Constraints**: Australian tax domain only. Current FY (2025-26) only. Single entity per plan (no group structures). One plan per client per FY.
**Scale/Scope**: 5-10 practice trials during April-June 2026 EOFY season. ~50-100 tax plans across trial practices.

## Key Design Decisions

### D1: Standalone Agent with Tool-Use (not Orchestrator Extension)

The AI scenario modelling uses a standalone `TaxPlanningAgent` class with its own router endpoints and direct Claude API calls, rather than routing through the existing `MultiPerspectiveOrchestrator`.

**Why**: The orchestrator is designed for single-turn, prompt-only general queries. Tax planning requires multi-turn conversation with persistent context and — critically — deterministic tax calculations via tool-use. The AI must call `calculate_tax_position` as a tool to produce accurate numbers, not generate them from the LLM.

```
User message → TaxPlanningAgent
  ├── Build system prompt (financial context + conversation history + scenario summaries)
  ├── Call Claude with tools: [calculate_tax_position, compare_scenarios]
  ├── Claude generates strategy analysis + calls calculate_tax_position tool
  ├── Tool returns deterministic tax figures
  ├── Claude incorporates figures into response
  ├── Parse response → create TaxScenario records
  └── Stream response to frontend via SSE
```

### D2: JSONB Financials on Tax Plan (not Separate Table)

Financial data is stored as a JSONB column on `tax_plans` rather than in a separate `tax_plan_financials` table.

**Why**: Financials are always 1:1 with the plan — there's no independent lifecycle, no separate access pattern, and no case where you'd query financials without their parent plan. A JSONB column eliminates a join and simplifies the data model without sacrificing queryability (PostgreSQL JSONB operators available if needed).

### D3: Database-Stored Tax Rates (not Hardcoded)

All Australian tax rates, brackets, thresholds, and offsets are stored in the `tax_rate_configs` table, seeded via Alembic migration.

**Why**: Tax rates change annually via legislation. The spec explicitly requires rates be updatable without code deployment. A database table with `(financial_year, rate_type)` as the unique key preserves historical rates (so past plans always reference their calculation basis) while allowing admin updates for new financial years.

### D4: Reuse Existing Xero P&L Pipeline

The tax planning module calls `XeroReportService.get_report(report_type="profit_and_loss", period_key="2025-FY")` to fetch client financials. No new Xero API integration.

**Why**: The complete P&L pipeline already exists — fetch, transform, cache with TTL. The `ProfitAndLossTransformer` already extracts revenue, other_income, cost_of_sales, operating_expenses, and net_profit. Reusing this avoids duplicating ~2,000 lines of tested Xero integration code.

### D5: Server-Side PDF with WeasyPrint

PDF export uses server-side HTML-to-PDF conversion with WeasyPrint and Jinja2 templates.

**Why**: Ensures consistent formatting regardless of client browser/OS. Allows injection of tenant branding (practice name, logo) from server-side data. Jinja2 templates are easy to customise without code changes.

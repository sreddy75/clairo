# Feature Specification: Trigger System

**Feature Branch**: `feature/017-trigger-system`
**Created**: 2025-12-31
**Status**: Draft
**Input**: Event/time-based trigger system for proactive insight generation

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Data-Driven Triggers (Priority: P1)

When financial data changes significantly, the system should automatically generate relevant insights without waiting for scheduled runs.

**Why this priority**: Data triggers provide immediate value by catching important changes (GST threshold crossings, cash flow alerts) as they happen, not hours or days later.

**Independent Test**: Can be fully tested by syncing Xero data that crosses a threshold and verifying an insight is generated within seconds.

**Acceptance Scenarios**:

1. **Given** a client's revenue is $70K, **When** a new invoice pushes revenue above $75K GST threshold, **Then** a GST Registration insight is generated immediately
2. **Given** a client has normal cash flow, **When** AR aging shows >$20K overdue invoices after sync, **Then** a Cash Flow Warning insight is generated
3. **Given** data quality was acceptable, **When** sync reveals >10 unreconciled transactions, **Then** a Data Quality insight is generated
4. **Given** a trigger fires, **When** an identical insight already exists (within 7 days), **Then** no duplicate insight is created

---

### User Story 2 - Time-Based Triggers (Priority: P2)

The system should run scheduled insight generation at appropriate intervals to catch issues proactively.

**Why this priority**: Time triggers ensure regular analysis even when data doesn't change dramatically, catching gradual trends and upcoming deadlines.

**Independent Test**: Can be tested by configuring a daily trigger and verifying insights are generated at the scheduled time.

**Acceptance Scenarios**:

1. **Given** daily insight generation is configured, **When** the scheduled time arrives, **Then** insights are generated for all active clients
2. **Given** weekly summary is configured, **When** Monday 9am arrives, **Then** a summary of the week's insights is available
3. **Given** quarterly BAS deadlines, **When** 14 days before deadline, **Then** BAS Deadline Reminder insights are generated for relevant clients
4. **Given** a time trigger runs, **When** the previous run was <1 hour ago, **Then** the trigger is skipped (debounce)

---

### User Story 3 - Event-Based Triggers (Priority: P2)

Specific business events should trigger targeted insight generation.

**Why this priority**: Event triggers enable real-time response to significant business events like new clients or BAS lodgements.

**Independent Test**: Can be tested by adding a new Xero connection and verifying welcome/onboarding insights are generated.

**Acceptance Scenarios**:

1. **Given** a new Xero connection is established, **When** initial sync completes, **Then** a "New Client Review" insight is generated
2. **Given** a BAS session is marked as lodged, **When** lodgement is recorded, **Then** a "Post-Lodgement Review" insight is generated
3. **Given** a client has an action item due in 3 days, **When** the daily check runs, **Then** an "Action Item Due Soon" reminder is generated
4. **Given** an event trigger fires, **When** the event has already been processed, **Then** idempotency prevents duplicate processing

---

### User Story 4 - Trigger Configuration & Monitoring (Priority: P3)

Accountants should be able to view and manage trigger configurations.

**Why this priority**: Configuration control is important but not essential for MVP - default triggers provide value without customization.

**Independent Test**: Can be tested by accessing the trigger admin page and viewing/modifying trigger settings.

**Acceptance Scenarios**:

1. **Given** an accountant accesses trigger settings, **When** they view the page, **Then** all configured triggers are listed with their status
2. **Given** a trigger is causing issues, **When** the accountant disables it, **Then** no new insights are generated by that trigger
3. **Given** triggers have run, **When** viewing trigger history, **Then** recent executions with counts are displayed
4. **Given** a trigger fails, **When** viewing trigger history, **Then** error details are visible for debugging

---

### Edge Cases

- What happens when multiple triggers fire for the same client simultaneously? (Batch and dedupe)
- How does the system handle trigger execution during maintenance? (Queue and retry)
- What if insight generation fails mid-trigger? (Partial completion tracking)
- How are triggers handled for inactive/disconnected clients? (Skip with warning)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST support data-change triggers that fire when financial metrics cross thresholds
- **FR-002**: System MUST support time-based triggers with cron-style scheduling
- **FR-003**: System MUST support event-based triggers for business events (new client, lodgement, etc.)
- **FR-004**: System MUST deduplicate insights within a configurable window (default 7 days)
- **FR-005**: System MUST track trigger execution history for monitoring
- **FR-006**: System MUST support enabling/disabling individual triggers
- **FR-007**: System MUST debounce rapid trigger executions to prevent spam
- **FR-008**: System MUST handle trigger failures gracefully with retry logic
- **FR-009**: System MUST log all trigger executions for audit purposes

### Key Entities

- **Trigger**: Configuration for when to generate insights (type, condition, target analyzer)
- **TriggerExecution**: Record of a trigger firing (timestamp, status, insights generated)
- **TriggerCondition**: Threshold or event criteria that activates a trigger

## Auditing & Compliance Checklist *(mandatory)*

### Audit Events Required

- [ ] **Authentication Events**: N/A - triggers run as system processes
- [x] **Data Access Events**: Triggers read client financial data to evaluate conditions
- [x] **Data Modification Events**: Triggers create insights as a result of evaluation
- [ ] **Integration Events**: N/A - uses already-synced data
- [x] **Compliance Events**: Some triggers relate to BAS deadlines and compliance

### Audit Implementation Requirements

| Event Type | Trigger | Data Captured | Retention | Sensitive Data |
|------------|---------|---------------|-----------|----------------|
| trigger.executed | Trigger runs | trigger_id, client_ids, insights_created | 5 years | None |
| trigger.failed | Trigger errors | trigger_id, error_message, stack_trace | 5 years | None |
| trigger.disabled | Admin disables trigger | trigger_id, disabled_by, reason | 7 years | None |
| insight.auto_created | Trigger creates insight | insight_id, trigger_id, client_id | 7 years | None |

### Compliance Considerations

- **ATO Requirements**: Trigger history may be relevant for demonstrating proactive compliance monitoring
- **Data Retention**: Trigger execution logs retained 5 years minimum
- **Access Logging**: System-level triggers logged with service account context

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Data triggers fire within 30 seconds of relevant data sync completing
- **SC-002**: Time triggers execute within 5 minutes of scheduled time
- **SC-003**: Event triggers fire within 10 seconds of event occurrence
- **SC-004**: Duplicate insight rate <5% (deduplication working effectively)
- **SC-005**: Trigger failure rate <1% under normal operation
- **SC-006**: System generates 5+ relevant insights per client per week via triggers
